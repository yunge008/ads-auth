# GMV 归因进度系统：审核与实施计划

> 文档状态：待外部审核，暂不作为上线实施依据。
>
> 编写日期：2026-07-11
>
> 本文整合截至 2026-07-11 已存在的归因代码和已确认业务规则。若本文与当前实现冲突，以本文“已确认规则”为下一轮改造目标；不得直接部署当前实现代替本文。

## 1. 目标与边界

系统用于将 TikTok GMV Max 素材数据归因到 BD、剪辑、商品卡或未归因桶，并提供：

- 管理员全量明细、任意时间汇总和按 VID 导出的可存档底表。
- 同事查询页的当前月进度和素材指标；只展示“设置 - 人员表”中启用的人员。
- 飞书中维护的绩效目标、站点交接、归因审查、统计快照和当前达人归属。
- 北京时间 08:00-23:00 每小时自动刷新；管理员可手动刷新，同事不可手动刷新但可见上次刷新时间。

不在本期范围：在线维护历史归因规则版本、用飞书直接承载大量 GMV 明细、以页面汇总反推底层归因。

## 2. 已确认的业务规则

### 2.1 归因优先级

每条广告汇总数据只能进入一个归因桶，顺序如下：

1. 商品卡：进入“商品卡”桶，不归属任何人员。
2. VID 强匹配：VID 在 BD 或剪辑登记中命中时，归该登记人。若多人登记同一 VID，产生审查项；无人工判定时归最后登记该 VID 的同事。若登记日期相同且没有更细时间，按来源表中较后的登记行作为最后登记，人工判定可覆盖。
3. BD 达人匹配：仅非 VID 强匹配的数据，按“人工判定别名 -> 同国家/站点的达人登记归属 -> VID 推断别名”匹配。
4. 未命中：进入“未归因”桶。

VID 强匹配不因站点交接或达人默认归属而改写。它表示该条素材由谁登记；统计维度仍取广告数据的实际国家/站点。

### 2.2 达人身份与跨国家同昵称

- 飞书登记源包含：国家/站点、昵称、用户名、BD/剪辑、登记日期、发样日期和 VID。
- 达人匹配的身份键必须带国家/站点，至少为 `国家/站点 + 匹配类型 + 归一化值`。
- 跨国家同昵称必须隔离，例如 PH 和 MX 同名达人不能共享归属。
- 同国家同昵称、但用户名不同的情况：仅在该广告行没有 VID 强匹配、只能按昵称匹配，且登记表也没有可用用户名时，进入未归因桶并生成审查项，等待人工处理；不得自动归属给任一同事。
- 昵称、用户名均做 NFKC、去首尾空格、空白折叠和小写化处理；空值或占位符不参与身份匹配。

### 2.3 站点交接

交接规则在“绩效配置表”中维护。示例：`PH | 阿木 | 李汝华 | 2026/7/1`。

- 以北京时间交接日 00:00 为边界。
- 对非 VID 强匹配素材：该站点交接日前归原 BD，交接日及之后归新 BD。
- 交接站点忽略原 BD 对达人形成的历史默认归属：交接后该站点全部达人素材均可归新 BD。
- 只有“绩效配置表 - 站点交接”区域存在有效登记的站点变更才是正式站点交接；未登记交接表的跨国出现一律视为短暂协助。短暂协助不改变达人默认归属；若某条素材有 VID 强匹配，则仍归该登记人，并按广告实际站点统计。
- 发布时间优先级：上传行内发布时间 > GMV 元数据发布时间 > 从 VID 推断。若仅能由 VID 推断且落在交接日正负 5 天内，写入审查提示。

### 2.4 汇率

- 汇率在前端“设置”中维护，不新增飞书汇率表。
- 当前预期主要是 THB 历史数据；仍需允许新增币种、启停和手动修改。
- 每一条最终归因明细必须保存原币种、原币 GMV/消耗、实际使用汇率、折算 USD 金额和计算时间。
- 不维护在线历史版本；用户每次计算后自行导出 Excel 快照存档。重新计算时使用当时前端配置的当前汇率。

### 2.5 2,000 USD 阈值

- 管理员页、导出底表、归因计算和飞书绩效统计记录一律保留全量数据，不应用 2,000 USD 展示过滤。
- 阈值仅适用于 BD 角色的同事页展示：某 BD 当月在某个实际站点的归因 GMV 小于 2,000 USD，且该站点未出现在该 BD 当月“绩效配置表”D 列的配置范围内时，该站点不在同事页展示。
- 已配置站点无论 GMV 是否达到 2,000 USD，均必须在同事页和绩效统计记录中展示；目标为 0 时仍展示数据，但 GMV 目标和 GMV 进度为空。
- 剪辑角色不受此阈值影响，按实际出现的全部国家/站点分行展示。
## 3. 飞书表设计

所有归因配置和回写均在“SKU商品信息匹配表”中完成。

### 3.1 绩效配置表

#### A:F 各同事绩效目标

| 列 | 字段 | 规则 |
| --- | --- | --- |
| A | 月份 | 接受 `YYYYMM` 或 `YYYY-MM`，系统统一为 `YYYY-MM`。 |
| B | 姓名 | 必须与设置 - 人员表中的名称一致。 |
| C | 角色 | `BD` 或 `剪辑`。 |
| D | 国家/站点 | 可用英文逗号分隔多个站点，如 `MX-AR,MX-NE,MX-SJ`；`ALL` 仅表示目标范围为全部站点。 |
| E | GMV目标 | USD。BD 可填；剪辑为 0，不设置 GMV 进度。 |
| F | 素材目标 | BD、剪辑均可填；0 表示不设素材进度。 |

合并站点目标是一条“目标组”，不是拆分后的多条独立目标。页面和统计记录按实际站点分行，但同目标组的每行展示同一目标和同一合计进度；这些重复目标值不可相加。

为使合并目标组可读，同事页和绩效统计记录必须额外显示四个目标组派生值：`当前总GMV`、`当前总GMV进度`、`当前总素材数量`、`当前素材进度`。它们按该行所属目标组的全部站点合计计算，并在组内每个站点行重复显示。单站点目标组的总值等于该站点值。GMV 目标为 0 时仍可显示当前总 GMV，但 GMV 进度为空；素材目标为 0 时素材进度为空。

#### H:L 站点交接

| 列 | 字段 | 规则 |
| --- | --- | --- |
| H | 国家/站点 | 单个站点。 |
| I | 原BD | 交接前负责人员。 |
| J | 新BD | 交接后负责人员。 |
| K | 交接日期 | 有效日期，按北京时间解释。 |
| L | 备注 | 交接背景、例外说明。 |

### 3.2 归因审查

采用审查日志结构：

`审查ID｜类型｜主体｜候选人员及默认归因｜证据摘要｜状态｜人工判定人员｜人工备注｜判定时间｜系统结果`

| 字段 | 含义 |
| --- | --- |
| 审查ID | 同一业务问题的稳定键，用于关联多次判定日志。 |
| 类型 | VID 重复登记、同国同昵称无用户名、昵称归属冲突、保护期争议、交接日附近等。 |
| 主体 | 被审查的 VID、昵称或交接事件。 |
| 候选人员及默认归因 | 可选归属人员，以及系统在无人判定时采用的结果。 |
| 证据摘要 | 登记日期、来源表、VID 数量、发布时间来源等。 |
| 状态 | 待处理、已处理、忽略。 |
| 人工判定人员 | 人工指定的最终人员；填“忽略”表示保持默认或不归因。 |
| 人工备注 | 判定原因。 |
| 判定时间 | 系统记录本次人工判定被采纳的时间。 |
| 系统结果 | 已采纳、未采纳、待同步等系统反馈。 |

审查表不是日常全量审核表，只记录会改变归因结果的例外。每次人工判定在此表追加一条审查日志；相同审查ID允许出现多条历史日志，系统按最新已采纳的判定作为当前有效判定。

人工判定的落地规则：

1. 系统将当前有效判定写入去重后的人工归因规则，去重键为 `国家/站点 + 匹配类型 + 归一化键`。
2. 对受影响的当前计算批次重新归因或定向更新。按 VID 汇总的归因明细对其唯一键只保留一条记录，原地更新归因人员、角色、归因来源和人工判定标记，绝不追加第二条 GMV 明细。
3. 完成月如需采纳新的人工判定，由管理员明确执行该月重新计算；导出的旧 Excel 仍是人工保存的历史快照。
### 3.3 绩效统计记录

按“计算批次 x 数据月份 x 人员/桶 x 实际国家/站点”追加快照：

`记录日期｜数据月份｜国家/站点｜姓名/桶｜角色｜归因GMV USD｜消耗 USD｜订单数｜目标GMV USD｜进度｜VID匹配GMV USD｜昵称匹配GMV USD｜商品卡GMV USD｜未归因GMV USD｜当月发样数量｜当月发样回收数量｜素材回收率｜当月VID登记素材数量｜备注｜当前总GMV USD｜当前总GMV进度｜当前总素材数量｜当前素材进度`

- “记录日期”是本次计算完成时间（北京时间）；“数据月份”是报表统计月份。
- 国家/站点始终使用广告数据的实际站点。剪辑也必须按站点分行，不写 `ALL` 汇总行。
- 归因 GMV、消耗和订单为该行人员/桶在该站点的实际值，统一折算 USD。
- 合并站点目标的目标和进度在各子站点行重复展示，只用于阅读，不得对这些列求和。
- 当前总 GMV、当前总素材数量和对应进度是目标组口径：同一组内各站点行显示同一数值，不得按行求和。
- 剪辑的目标 GMV、进度为空；仅展示其 VID 强匹配所得的 GMV/消耗/订单和素材数。
- 剪辑也按实际站点分行；若其配置了素材目标（例如 `ALL / 400`），当前总素材数量和当前素材进度按该目标组汇总后重复展示在各站点行。
- 商品卡、未归因是桶行，不设目标与素材指标。

BD 素材指标定义：

1. 当月发样数量：筛选 B 发样日期在当月后，按 `同事 + 国家/站点 + 达人键 + SKU(K列)` 合并去重。
2. 当月发样回收数量：在上述范围内进一步筛选 P 列存在有效 VID，再使用相同去重键计数。
3. 素材回收率：回收数量 / 发样数量；分母为 0 时为空。
4. 当月 VID 登记素材数量：P 有有效 VID 且 N 登记日期在当月，建议按标准化 VID 去重计数。

达人键为 `国家/站点 + (用户名优先；用户名为空时为标准化昵称)`，与达人身份匹配口径一致，防止跨国家同名达人被错误合并。用于去重的站点、达人或 SKU 为空时不可将所有空值合并为一条，应保留原始行并标记数据质量问题。

### 3.4 归因记录

列结构：

`类型｜国家/站点｜用户名｜昵称｜归一化键｜当前归属BD｜最后登记日期｜归因来源｜交接提示｜备注`

这是“当前状态表”，不是历史记录表：

- 它显示某个国家/站点下，达人用户名或昵称当前的默认归属。
- 当前归属由建联保护期、人工判定或 VID 推断得出。
- 站点交接只在计算某条素材的最终归因时覆盖默认归属；归因记录不改写历史交接前后的状态。
- 系统覆盖写该表，人工不要直接改其结果。人工修正走“归因审查”，再写入可追溯的人工归因规则。
- 历史结果以“按 VID 汇总的归因明细快照”为准。

## 4. 管理员与同事页面

### 管理员页

- 可选择任意时间范围；展示全量 GMV、消耗、订单、商品卡、未归因和各归因逻辑。
- 可查看每行的月度拆分。
- 不绑定 GMV KPI，不应用 2,000 USD 过滤，不因人员是否启用而隐藏历史数据。
- 可手动刷新、同步配置、处理审查项、导出按 VID 汇总的归因明细。

### 同事查询页

- 仅显示设置 - 人员表中启用的同事。
- 展示当前月份所有启用人员的站点分行统计和进度，并显示上次成功刷新时间。
- 不提供手动刷新入口。
- BD：归因 GMV/KPI 进度、发样数、回收数、回收率、VID 登记素材数。
- 剪辑：按站点展示 VID 强匹配归因 GMV/消耗/订单与 VID 登记素材数；无 GMV KPI。

## 5. 唯一统计底表与导出

系统每次完成计算后，必须生成可下载的“按 VID 汇总归因明细”。它是管理员全量数据和所有后续透视统计的唯一底表。

推荐唯一粒度：

`计算批次 + 数据月份 + VID + 国家/站点 + 广告账号 + 内容类型 + 原币种`

不能只以 VID 作为唯一键，因为同一 VID 可能在不同国家、广告账号、内容类型或币种中出现。

最低字段：

`计算批次ID、计算完成时间、数据月份、VID、广告账号、达人昵称、国家/站点、内容类型、发布时间、发布时间来源、活跃天数、原币种、原币GMV、原币消耗、汇率、GMV USD、消耗 USD、订单数、归因桶、归因人员、角色、归因逻辑、达人默认归属、交接是否生效、目标组、是否计绩效、备注`

该表必须包含低于 2,000 USD 的数据。每个唯一键只保留一条当前生效记录；人工判定不会向该表追加第二条记录，只会更新该条记录的归因人员和归因来源。管理员可直接下载为 Excel；用户自行存档的文件即为该次 GMV 状态快照。

## 6. 数据与性能架构

### 6.1 推荐计算链路

```text
TikTok 日明细 + 广告账号站点
        + 飞书达人登记/人员表
        + 前端汇率/绩效配置/站点交接/人工判定
                         |
                         v
              按 VID 汇总的归因计算批次
                         |
          +--------------+--------------+
          |              |              |
          v              v              v
    管理员全量视图   同事当前月视图   飞书统计快照/Excel导出
```

必须以“计算批次明细”作为单一事实来源，再派生页面、飞书快照和导出，不允许三个出口各自重新归因。

### 6.2 固化与刷新

- 已完成月份：每月只保留一份基础归因明细表和一份结果表，前端整月查询直接读取结果表。管理员明确发起“重新计算该月”时才替换这两份当前在线结果；用户已导出的 Excel 负责人工历史存档。
- 当前月：08:00-23:00 每小时计算并覆盖“当前有效批次”；管理员手动刷新可立即创建新批次。
- 任意时间段：优先拼接已固化月度批次；若选区间没有匹配的固化数据，再走全量计算入口。
- 每个批次记录配置快照或至少记录使用的汇率和规则版本摘要，保证导出可解释。

### 6.3 避免卡顿和冗余

- 页面请求只读取最近成功批次及其聚合，不在每次打开页面时重新扫描 `gmv_max_vid_daily` 并执行归因。
- 当前月刷新按月计算，完成后原子切换“当前有效批次”；读取侧始终读取成功批次，避免看到半成品。
- 后端明细查询按月份、国家/站点、VID 建索引；大数据量使用键集分页，不使用深 OFFSET 分页。
- 飞书写入按批次追加“绩效统计记录”，按页分块写入；归因记录是当前状态镜像，覆盖写。
- 管理员多月汇总直接聚合批次明细或月度预聚合，不重复调用 TikTok API。

## 7. 当前项目进度与代码逻辑

### 7.1 项目轮廓

- 前端：React/TanStack Start，现有“GMV 归因”和“GMV 归因·管理”两个页面入口；设置页已有人员启用状态。
- 后端：Supabase Edge Functions；TikTok GMV Max 日数据已持续写入 Postgres。
- 数据源：`gmv_max_vid_daily` 保存日粒度广告数据，`gmv_max_vid_meta` 保存可获得的 VID 发布时间，人员/素材登记来自飞书。
- 当前归因代码已在工作区完成但尚未完成数据库 migration、函数部署和人工验收；因此当前状态是“原型代码完成，业务规则待本计划改造后验证”，不是可上线状态。

### 7.2 现有归因代码的工作方式

当前实现的计算链路为：

```text
飞书建联/归档/剪辑登记
  -> 同步为达人登记原始表
  -> 解析达人默认归属和保护期
  -> 月度聚合 TikTok 日明细
  -> 商品卡/VID/昵称三级归因
  -> 管理员或同事页面汇总
  -> 飞书回写和 Excel 上传结果
```

现有模块职责：

| 模块 | 当前职责 |
| --- | --- |
| 达人同步 | 读取 BD 建联表、授权归档和剪辑表，生成达人登记原始记录，并计算昵称/用户名的默认归属。 |
| 月度聚合 | 从 TikTok 日明细按 `VID + 广告账号 + 内容类型 + 国家 + 币种` 聚合 GMV、消耗、订单和活跃天数。 |
| 归因引擎 | 商品卡单列；VID 强匹配到 BD/剪辑；未命中 VID 的 BD 数据走人工别名、登记归属、VID 推断别名；最后是未归因。 |
| 交接处理 | 当前只在昵称路径中按发布时间调整归属，发布时间可来自上传字段、VID 元数据或 VID 推断。 |
| 汇总 | 生成人员、国家/站点、商品卡、未归因和归因逻辑桶的 GMV/消耗/订单汇总。 |

## 8. 当前实现状态与差异

当前工作区已存在一套未部署验证的归因实现，包含：归因引擎、月度 RPC 聚合、飞书同步、Excel 上传、两张页面和数据库 migration。其大方向可复用，但以下差异必须在部署前修正：

| 范围 | 当前实现 | 本计划要求 |
| --- | --- | --- |
| 达人归属唯一键 | `key_type + match_key`，昵称全局匹配 | `国家/站点 + key_type + match_key`，跨国隔离。 |
| 目标唯一键 | `month + staff_name + role` | 支持目标组及多个站点；不得将合并站点目标重复相加。 |
| 飞书 sheet | 旧方案要求单独的 GMV目标、站点交接、归因进度、达人归因表 | 统一改为 SKU商品信息匹配表中的绩效配置表、归因审查、绩效统计记录、归因记录。 |
| 剪辑展示 | 旧逻辑存在 `ALL` 汇总预期 | 按广告实际国家/站点分行，不写 ALL 记录。 |
| 非 USD | 当前仅告警，不进入桶和进度 | 按前端汇率折 USD 后进入完整归因与统计。 |
| 2,000 USD | 同事页低于阈值的站点被隐藏 | 仅 BD 同事页中“未配置站点 + 单站点归因GMV小于2,000 USD”不展示；剪辑、管理页、绩效统计记录和导出均全量保留。 |
| 归因记录 | 当前仅输出镜像 | 保持当前状态镜像；人工修改必须经审查规则。 |
| 刷新方式 | 打开月报可能触发重新聚合/归因 | 按计算批次固化，页面只读取批次。 |
| 导出 | 尚未形成唯一的 VID 底表 | 导出完整按 VID 汇总归因明细，支持透视复算。 |

## 9. 实施计划

### 阶段 0：审核与数据准备

1. 审核本文，确认所有“已确认规则”无冲突。
2. 在“SKU商品信息匹配表”准备绩效配置表、归因审查、绩效统计记录、归因记录及表头。
3. 确认源登记表中 B、K、N、P 列的实际表头与语义：K 必须可作为“同事 + 站点 + 达人 + SKU”发样去重的一部分。
4. 整理有效的站点代码、人员名称、角色名称，确保与设置 - 人员表一致。

验收：用 3-5 条覆盖跨国同昵称、同国同昵称、VID 强匹配、交接前后、THB 的样例人工算出预期结果。

### 阶段 1：数据模型与配置同步

1. 新建 migration，不修改历史 migration。
2. 将达人归属、人工别名、目标组/目标站点映射、计算批次和批次明细模型补齐。
3. 修改飞书同步，读取绩效配置表 A:F 与 H:L，写入/读取新 sheet 名和列布局。
4. 在设置页增加汇率配置，含币种、兑 USD 汇率、启用状态和更新时间。

验收：同一昵称在 PH/MX 分属不同 BD 时互不影响；合并站点目标只计算一次；THB 行写入正确汇率和 USD 金额。

### 阶段 2：归因引擎与指标

1. 将所有达人匹配映射改为国家/站点维度。
2. 固化 VID 强匹配、交接覆盖、人工判定和审查项的优先级。
3. 实现 BD 的发样/回收/素材指标，以及剪辑按站点的 VID 指标。
4. 将 2,000 USD 改为绩效标记，不再作为数据过滤条件。

验收：对阶段 0 样例逐条核验归因桶、归因人员、站点、汇率、指标和进度。

### 阶段 3：批次、导出与页面

1. 实现月度固化批次、当前月有效批次和任意时间段回退全量计算。
2. 从批次明细生成管理员全量页、同事当前月页、飞书绩效统计记录和 Excel 导出。
3. 管理员导出完整按 VID 汇总明细；同事页只读并显示上次刷新时间。
4. 管理员页支持多月汇总和月度明细展开，不绑定 KPI。

验收：同一批次的管理员页、同事页、飞书统计记录和导出透视表可互相核对；低于 2,000 USD 的数据只影响绩效标记，不消失。

### 阶段 4：调度、性能和上线验证

1. 配置北京时间 08:00-23:00 每小时任务，仅刷新当前月；管理员手动刷新走同一批次流程。
2. 增加刷新状态、失败原因、最近成功时间与并发锁，防止重复运行。
3. 针对大月数据测试索引、分页、飞书分块写入与导出内存。
4. 从空数据库重新拉取历史数据，核验 VID 发布日期和数据颗粒度。

验收：连续两次刷新结果幂等；刷新失败不替换上次成功批次；管理员导出与同月固化快照一致。

## 10. 上线前必须关闭的风险

1. K 列虽被业务定义为 SKU 去重维度，但必须确认不会将同一同事、站点、达人发出的两次独立同 SKU 样品错误合并；如会发生，应增加发样单号或发样日期等更细粒度字段。
2. 目标组的重复展示不能进入求和逻辑，否则多站点目标会放大。
3. 交接边界必须统一北京时间和“交接日及之后归新 BD”的包含关系。
4. 以昵称匹配时必须强制带国家/站点，不能回退到全局昵称查询。
5. 汇率变更后允许重新计算当前数据，但已导出的 Excel 是唯一人工存档；导出必须写明汇率。
6. 归因记录是当前状态，不可被当作历史事实或手工主数据。
7. 当前已有代码尚未完成部署验证，且与本文存在字段和规则差异；在完成阶段 1-4 前不得按“代码完成”视为可上线。

## 11. 审核问题清单

请审核者重点确认：

1. 已配置站点无条件展示、未配置站点才按 2,000 USD 计入同事绩效的口径，是否符合考核规则。
2. 站点交接是否只覆盖非 VID 强匹配素材，符合“协助人员”场景。
3. 同国家同昵称无用户名时默认归因的风险是否可接受，是否需强制审查提示。
4. `同事 + 站点 + 达人 + SKU` 是否足以代表一次发样，是否需要额外的发样单号。
5. 剪辑按站点拆行后的素材数和归因 GMV 是否符合绩效展示预期。
6. 归因明细的唯一粒度是否能覆盖同 VID 跨账号、跨国家、跨币种的情况。
7. 当前月每小时刷新、完成月固化、任意时间段回算的成本和一致性是否可接受。
